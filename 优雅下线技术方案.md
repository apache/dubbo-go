# 优雅下线技术方案

## 一、概述

### 1.1 与需求文档背景和目标的对应

**需求文档 1.1 背景问题与解决方案**：

| 背景问题 | 技术方案解决方案 |
|----------|-----------------|
| 过度依赖注册中心 | 被动通知 + 本地缓存兜底 |
| 缺乏主动通知 | 预留接口（后续迭代实现） |
| Consumer 无感知 | 响应携带 Closing 标记 + 本地 closingInvokers 缓存 |
| 缺乏 K8s 集成 | 用户自行实现 HTTP 端点（框架无内置） |

**需求文档 1.2 目标**：
1. 降低注册中心依赖：实现主动通知机制，注册中心作为兜底
2. Consumer 端感知：Consumer 能够感知 Provider 下线并移除不可用实例
3. 支持 K8s 探针（可选）
4. 保持兼容性

**技术方案实现情况**：

| 目标 | 实现情况 | 说明 |
|------|----------|------|
| 降低注册中心依赖 | 🔶 被动通知为主 | 经调研，协议层无获取连接列表接口，主动通知作为后续迭代 |
| Consumer 端感知 | ✅ 已实现 | 响应携带 Closing 标记 + 本地缓存 |
| K8s 探针 | ⏸️ 用户自行实现 | dubbo-go 无内置 HTTP Server |
| 保持兼容性 | ✅ 已设计 | 向后兼容、渐进式迁移 |

### 1.2 本期实现功能

基于需求文档确定的范围，本期实现以下功能：

1. **Provider 端**：
   - Closing 状态标记
   - 被动通知（响应携带 Closing 标记）
   - 主动通知（⚠️ 作为后续迭代，本期仅预留接口）
2. **Consumer 端**：感知 Provider 下线并移除不可用实例

### 2.3 Consumer Filter 扩展

**需求来源**：需求文档提到"Consumer 端 Filter 只做计数，无法感知 Provider 下线"，需要扩展现有 Filter。

现有 `consumerGracefulShutdownFilter` 只负责计数：
- `Invoke()`: 增加 `ConsumerActiveCount`
- `OnResponse()`: 减少 `ConsumerActiveCount`

扩展后增加以下功能：
- 检测响应中的 Closing 标记
- 维护 closingInvokers 本地缓存
- 发起请求前检查 Invoker 是否正在关闭

```go
// filter/graceful_shutdown/consumer_filter.go

const (
    GracefulShutdownClosingKey = "closing"
)

type consumerGracefulShutdownFilter struct {
    shutdownConfig *global.ShutdownConfig
    // 新增：存储正在关闭的 Invoker
    closingInvokers sync.Map // map[string]time.Time (url key -> expire time)
}

func (f *consumerGracefulShutdownFilter) Invoke(ctx context.Context, invoker base.Invoker, invocation base.Invocation) result.Result {
    // 新增：检查 Invoker 是否正在关闭
    if f.isClosingInvoker(invoker) {
        return &result.RPCResult{Err: errors.New("provider is closing")}
    }
    f.shutdownConfig.ConsumerActiveCount.Inc()
    return invoker.Invoke(ctx, invocation)
}

func (f *consumerGracefulShutdownFilter) OnResponse(ctx context.Context, result result.Result, invoker base.Invoker, invocation base.Invocation) result.Result {
    f.shutdownConfig.ConsumerActiveCount.Dec()

    // 新增：检测响应中的 Closing 标记
    if f.isClosingResponse(result) {
        f.markClosingInvoker(invoker)
    }

    return result
}

// isClosingInvoker 检查 Invoker 是否在 closing 列表中
func (f *consumerGracefulShutdownFilter) isClosingInvoker(invoker base.Invoker) bool {
    key := invoker.GetURL().String()
    if expireTime, ok := f.closingInvokers.Load(key); ok {
        // 检查是否过期
        if time.Now().Before(expireTime.(time.Time)) {
            return true
        }
        f.closingInvokers.Delete(key)
    }
    return false
}

// isClosingResponse 检查响应是否携带 Closing 标记
func (f *consumerGracefulShutdownFilter) isClosingResponse(result result.Result) bool {
    if result != nil && result.Attachments() != nil {
        if v, ok := result.Attachments()[GracefulShutdownClosingKey]; ok {
            if v == "true" {
                return true
            }
        }
    }
    return false
}

// markClosingInvoker 标记 Invoker 为正在关闭
func (f *consumerGracefulShutdownFilter) markClosingInvoker(invoker base.Invoker) {
    key := invoker.GetURL().String()
    // 设置过期时间（例如 30 秒后过期）
    expireTime := time.Now().Add(30 * time.Second)
    f.closingInvokers.Store(key, expireTime)
}
```

### 3.3 R6 移除不可用实例（详细设计）

**需求**：需求文档 R6 要求"当感知到 Provider 下线后，将其从可用实例列表中移除。"

**实现说明**：

需求文档中的"移除"通过以下两种机制实现：
1. **本地缓存标记**：通过 `closingInvokers` 本地缓存标记该 Invoker 不可用
2. **注册中心通知**：Provider 反注册后，注册中心通知 Consumer 移除该实例

**说明**：
- 技术方案不直接操作注册中心的实例列表（那是注册中心的职责）
- 通过本地缓存 + 注册中心通知的双重机制实现"移除"效果
- 本地缓存作为临时兜底，注册中心通知是正式方式

**实现方案**：

```go
// filter/graceful_shutdown/consumer_filter.go

const (
    // Closing Invoker 过期时间（可配置）
    defaultClosingInvokerExpireTime = 30 * time.Second
)

type consumerGracefulShutdownFilter struct {
    shutdownConfig *global.ShutdownConfig
    closingInvokers sync.Map // map[string]time.Time (url key -> expire time)
}

// markClosingInvoker 标记 Invoker 为正在关闭
func (f *consumerGracefulShutdownFilter) markClosingInvoker(invoker base.Invoker) {
    key := invoker.GetURL().String()
    expireTime := time.Now().Add(defaultClosingInvokerExpireTime)
    f.closingInvokers.Store(key, expireTime)
    logger.Infof("Marked invoker as closing: %s, will expire at %v", key, expireTime)
}

// removeClosingInvoker 移除已过期的 Closing Invoker
func (f *consumerGracefulShutdownFilter) removeClosingInvoker(key string) {
    f.closingInvokers.Delete(key)
}

// cleanupExpiredClosingInvokers 清理所有已过期的 Closing Invoker
// 定期调用或在检测时调用
func (f *consumerGracefulShutdownFilter) cleanupExpiredClosingInvokers() {
    now := time.Now()
    f.closingInvokers.Range(func(key, value any) bool {
        if expireTime, ok := value.(time.Time); ok {
            if now.After(expireTime) {
                f.closingInvokers.Delete(key)
                logger.Infof("Removed expired closing invoker: %s", key)
            }
        }
        return true
    })
}
```

**与注册中心配合**（需求文档 R6 要求）：

1. **依赖注册中心的事件**：
   - 当 Provider 执行 `destroyRegistries()` 反注册时，注册中心会通知所有订阅的 Consumer
   - Consumer 端的注册中心监听器会收到 Provider 下线事件
   - 注册中心自动将该 Provider 从可用实例列表中移除

2. **本地缓存兜底**：
   - 如果注册中心通知未及时到达（网络延迟、丢包等），Consumer 本地的 `closingInvokers` 缓存可以阻止发往已下线 Provider 的请求
   - 这是双重保障机制：注册中心 + 本地缓存

3. **过期机制**：
   - Closing Invoker 设置过期时间（如 30 秒），过期后自动失效
   - 如果 Provider 重新上线，Consumer 可以再次发现并使用该实例

4. **与注册中心实例管理交互**：
   - 不直接操作注册中心的实例列表
   - 依赖注册中心的通知机制来更新实例列表
   - 本地 `closingInvokers` 仅作为临时缓存

**关键约束**：
- 移除后不应立即删除，需设置过期时间
- 过期时间需要可配置
- 需要定期清理过期的 Closing Invoker

var (

        "dubbo_provider_active_requests",

    )

        "Number of invokers marked as closing",
    )

// 更新 Provider 端指标
func updateProviderMetrics(shutdown *global.ShutdownConfig) {
    providerActiveRequests.Set(float64(shutdown.ProviderActiveCount.Load()))
    if shutdown.Closing.Load() {
        providerClosingState.Set(1)
    } else {
        providerClosingState.Set(0)
    }
}

// 更新 Consumer 端指标
func updateConsumerMetrics(filter *consumerGracefulShutdownFilter) {
    var count int64
    filter.closingInvokers.Range(func(key, value any) bool {
        count++
        return true
    })
    consumerClosingInvokers.Set(float64(count))
}

## 七、错误处理与容错

### 7.1 需求来源

需求文档 R1 明确要求：
> 任意步骤失败不影响整体流程继续
> 每个步骤超时后强制进入下一步

### 7.2 各步骤错误处理策略

| 步骤 | 失败处理 | 超时处理 |
|------|----------|----------|
| 1. 标记 Closing 状态 | 记录警告，继续下一步 | 不适用 |
| 2. 主动通知长连接 | 记录警告，继续下一步（有注册中心兜底） | 强制进入下一步 |
| 3. 反注册注册中心 | 记录警告，继续下一步 | 不适用 |
| 4. 拒绝新请求 | 记录警告，继续下一步 | 不适用 |
| 5. 等待请求完成 | - | 强制进入下一步 |
| 6. 等待响应完成 | - | 强制进入下一步 |
| 7. 销毁协议 | 记录警告，继续下一步 | 不适用 |
| 8. 执行回调 | 记录警告，继续下一步 | 不适用 |

### 7.3 核心原则

**任意步骤失败不影响整体流程继续**

```go
// graceful_shutdown/shutdown.go

func beforeShutdown(shutdown *global.ShutdownConfig) {
    // 步骤 1：标记 Closing 状态
    shutdown.Closing.Store(true)

    // 步骤 2：主动通知（失败不影响后续流程）
    if err := notifyLongConnectionConsumers(); err != nil {
        logger.Warnf("Graceful shutdown: notify failed, err=%v, continuing...", err)
    }

    // 步骤 3：反注册（失败不影响后续流程）
    if err := destroyRegistries(); err != nil {
        logger.Warnf("Graceful shutdown: destroy registries failed, err=%v, continuing...", err)
    }

    // 步骤 4-5：拒绝新请求 + 等待完成（有超时控制）
    waitAndAcceptNewRequests(shutdown)

    // 步骤 6：等待响应完成（有超时控制）
    waitForSendingAndReceivingRequests(shutdown)

    // 步骤 7：销毁协议（失败不影响后续流程）
    if err := destroyProtocols(); err != nil {
        logger.Warnf("Graceful shutdown: destroy protocols failed, err=%v, continuing...", err)
    }

    // 步骤 8：执行回调（失败不影响后续流程）
    executeCustomCallbacks()
}
```

### 7.4 主动通知超时处理

```go
func notifyLongConnectionConsumers() error {
    stepTimeout := getStepTimeout()
    done := make(chan struct{})

    go func() {
        // 执行主动通知
        notifyTripleConsumers()
        notifyGrpcConsumers()
        close(done)
    }()

    select {
    case <-done:
        // 通知完成
        return nil
    case <-time.After(stepTimeout):
        // 超时，强制进入下一步
        logger.Warn("Graceful shutdown: notify timeout, continuing...")
        return nil
    }
}
```

## 八、兼容性设计

### 7.1 向后兼容

1. **配置兼容**：
   - 所有新字段都有默认值，不影响现有配置
   - 新增 `Closing` 字段默认为 false，不改变现有行为

2. **行为兼容**：
   - 被动通知通过响应 Attachment 携带，现有 Consumer 不感知
   - 只有感知到 Closing 标记的 Consumer 才会执行跳过逻辑

### 7.2 渐进式迁移

1. **Phase 1**：Provider 单方面升级
   - Provider 升级后，被动通知生效
   - 未升级的 Consumer 不受影响

2. **Phase 2**：Consumer 升级
   - Consumer 升级后，可以感知 Closing 标记
   - 开始跳过正在关闭的 Provider

## 十、风险与注意事项

### 10.1 已知约束（需求文档 六）

**需求文档明确的不支持场景**：

| 场景 | 说明 |
|------|------|
| 不支持中止正在处理的请求 | 优雅下线过程中，不会主动中止正在处理的请求，等待其自然完成 |
| 不支持请求重定向 | 不会自动将请求重定向到其他 Provider，由 Consumer 端决定如何处理 |

### 10.2 风险提示（需求文档 六）

| 风险 | 说明 |
|------|------|
| 少量失败请求 | 如果 Consumer 未及时响应通知，仍可能产生少量失败请求 |
| 请求被中断 | 超时后的强制下线可能导致正在处理中的请求被中断 |

### 10.3 技术方案风险

1. **Consumer 端缓存过期**：当前设计使用 30 秒过期时间，需要根据实际场景调整
2. **注册中心通知**：仍有注册中心作为兜底，被动通知失败不影响整体流程
3. **向后兼容**：修改不影响现有业务逻辑
