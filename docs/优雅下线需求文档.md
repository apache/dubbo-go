# 优雅下线需求文档

## 一、背景与目标

### 1.1 背景

当前 dubbo-go 的优雅下线机制存在以下问题：

1. **过度依赖注册中心**：流量摘除完全依赖注册中心通知，存在通知延迟或丢失风险
2. **缺乏主动通知**：没有通过长连接主动通知 Consumer
3. **Consumer 无感知**：Consumer 端 Filter 只做计数，无法感知 Provider 下线

### 1.2 目标

1. **降低注册中心依赖**：实现主动通知机制 + 被动通知机制，注册中心作为兜底
2. **Consumer 端感知**：Consumer 能够感知 Provider 下线并移除不可用实例
3. **保持兼容性**：渐进式迁移，不影响现有业务

#### R2：主动通知机制

**需求描述**：对于 Triple 等依赖长连接的协议，通过已建立的 TCP 长连接，直接通知 Consumer 当前 Provider 即将下线。

**针对协议**：Triple、gRPC、Dubbo 等长连接协议

**实现要求**：
- 获取当前维护的长连接列表
- 向每个连接发送关闭通知
- 设置超时时间，避免网络原因导致该步骤耗时太久
- 支持重试机制

**重试机制**：
- 长连接的保活机制并不能保证当前正在维护着的长连接就是活跃的
- 因此可能会通知失败，需要进行重试
- 在超时时间内，若收到和长连接个数正确的响应，则直接继续下一步
- 超时时间到之后，不再进行重试，直接进入下一步

**超时设计**：
- 主动通知步骤超时时间可配置，默认复用 StepTimeout（3s）
- 该步骤的超时时间可以和等待注册中心的下线通知下发的时间相关

**兜底机制**：
- 通知失败不影响后续流程（有注册中心作为保底）

#### R3：被动通知机制

**需求描述**：在 Provider 响应中携带 Closing 标记，Consumer 可据此感知 Provider 下线。

**实现方式**：
- 使用 Response 的 Attachment 携带 `closing=true` 标记
- 通过 Provider Filter 的 `OnResponse()` 函数携带标记
- 所有在 Closing 状态下发出的响应都应携带该标记

#### R4：请求计数与滑动窗口

**需求描述**：准确判断当前是否还有进行中的请求。

**问题背景**：
- 原本的逻辑是以 10ms 为间隔轮询 counter
- 在等待未超时的情况下，一旦发现 counter 为 0，就结束该步骤
- 但这样会有问题：在 counter 刚好为 0 的那一瞬间，可能刚好有请求刚返回或刚到达

**算法要求**：
- 维护一个请求进入的 counter
- 维护一个最近请求的时间戳
- 使用滑动窗口算法：counter == 0 且 最近请求时间超过窗口期 才认为请求处理完成

**滑动窗口参数**：
- 默认值：3s
- 轮询间隔：10ms

### 2.2 Consumer 端需求

#### R5：感知 Provider 下线

**需求描述**：Consumer 不主动触发下线，只负责接收 Provider 的下线通知并处理，避免继续向已下线的 Provider 发请求。

**感知方式**（两种都需要实现）：

| 方式 | 描述 |
|------|------|
| 被动感知 | Consumer 的 Filter 在处理 Provider 响应时，OnResponse 中检测到响应中携带的 "Closing" 标记（Attachment 载体） |
| 主动感知 | Consumer 的长连接接收到 Provider 主动发送的 Closing 下线请求 |

#### R6：移除不可用实例

**需求描述**：当感知到 Provider 下线后，将其从可用实例列表中移除。

**实现方式**：
- 当 Filter 检测到 Closing 标记时，将目标实例标记为不可用
- 当长连接接收到关闭请求时，将目标实例标记为不可用

**关键约束**：
- 移除后不应立即删除，需支持后续可能的恢复（如 Provider 重新上线）
- 设置过期时间，过期后自动恢复
- 需要与注册中心的实例管理机制配合

## 三、主动通知与被动通知结合

### 3.1 采用主动通知的原因

- 可以迅速的对当前连接着的客户端做出通知
- 不必等待注册中心的通知，注册中心的通知可能不稳定或者被降级导致不够及时

### 3.2 采用被动通知的原因

- 对于上游目前活跃的客户端：在一个等待期内极大概率会接收到它的请求，响应中携带标识就可以将其正确通知
- 对于上游目前不活跃的客户端：QPS 不高，在注册中心的反注册事件通知或者拉取最新配置之前，失败的请求不会很多

---

## 四、非功能性需求

### 4.1 兼容性

- 新增配置项应有合理默认值，不影响现有业务
- 原有配置结构保持不变，新配置作为扩展

### 4.2 可观测性

- 需要有日志记录优雅下线的关键步骤
- 关键指标建议暴露：
  - 进行中请求数
  - 已通知连接数
  - Closing 状态

---

## 五、已知约束

1. **不支持的场景**：
   - 不支持在优雅下线过程中中止正在处理的请求
   - 不支持请求重定向到其他 Provider（由 Consumer 端决定）

2. **风险提示**：
   - 如果 Consumer 未及时响应通知，仍可能产生少量失败请求
   - 超时后的强制下线可能导致正在处理中的请求被中断
