# 优雅下线需求文档

## 一、背景与目标

### 1.1 背景

当前 dubbo-go 的优雅下线机制（代码分析见附件）存在以下问题：

1. **过度依赖注册中心**：流量摘除完全依赖注册中心通知，存在通知延迟或丢失风险
2. **缺乏主动通知**：没有通过长连接主动通知 Consumer
3. **Consumer 无感知**：Consumer 端 Filter 只做计数，无法感知 Provider 下线
4. **缺乏 K8s 集成**：无法与容器健康检查配合

### 1.2 目标

1. **降低注册中心依赖**：实现主动通知机制，注册中心作为兜底
2. **Consumer 端感知**：Consumer 能够感知 Provider 下线并移除不可用实例
3. **支持 K8s 探针**（可选）：允许在优雅下线时关闭健康检查
4. **保持兼容性**：渐进式迁移，不影响现有业务

## 三、核心需求

### 3.1 Provider 端需求

#### R1：优雅下线流程

当进程收到终止信号（SIGTERM）时，执行以下步骤：

| 步骤 | 动作 | 超时控制 | 状态 |
|------|------|----------|------|
| 1 | 标记 Closing 状态 | - | 🔶 新增 |
| 2 | 被动通知（响应携带 Closing 标记） | - | 🔶 新增 |
| 3 | 反注册注册中心 | - | ✅ 已有 |
| 4 | 拒绝新请求 | - | ✅ 已有 |
| 5 | 等待进行中请求完成 | StepTimeout | ✅ 已有 |
| 6 | 等待发出的请求收到响应 | StepTimeout | ✅ 已有 |
| 7 | 销毁协议、释放资源 | - | ✅ 已有 |
| 8 | 执行回调 | - | ✅ 已有 |

**关键约束**：
- 总超时时间可配置，默认 60s
- 每个步骤超时后强制进入下一步
- 任意步骤失败不影响整体流程继续

#### R2：主动通知机制

**决策**：采用**被动通知 + 注册中心兜底**方案，主动通知作为后续迭代。

**原因**：
1. 协议层无获取连接列表的接口
2. 通知协议格式需要自定义开发
3. Consumer 端需要配合开发
4. 复杂度高，收益有限

**替代方案**：
- 被动通知：在响应中携带 Closing 标记
- 注册中心兜底：反注册后 Consumer 感知下线

**本需求范围**：仅实现被动通知 + 注册中心兜底。

#### R3：被动通知机制

**需求描述**：在 Provider 响应中携带 Closing 标记，Consumer 可据此感知 Provider 下线。

**实现方式**：
- 使用 Response 的 Attachment 携带 `closing=true` 标记
- 所有在 Closing 状态下发出的响应都应携带该标记

#### R4：请求计数与滑动窗口

**状态**：现有代码已有实现

现有代码 `graceful_shutdown/shutdown.go:173-179` 已实现滑动窗口逻辑：

```go
// 现有实现
for time.Now().Before(deadline) &&
    (shutdown.ProviderActiveCount.Load() > 0 ||
     time.Now().Before(shutdown.ProviderLastReceivedRequestTime.Load().Add(offlineRequestWindowTimeout))) {
    time.Sleep(10 * time.Millisecond)
}
```

**结论**：此需求已满足，无需额外开发。

### 3.2 Consumer 端需求

#### R5：感知 Provider 下线

**需求描述**：Consumer 需要感知到 Provider 的下线事件，避免将请求发送到已下线的 Provider。

**感知方式**（本期仅实现被动感知）：

| 方式 | 描述 | 优先级 | 状态 |
|------|------|--------|------|
| 被动感知 | 响应中携带 Closing 标记 | 高 | 🔶 本期实现 |
| 主动感知 | 长连接接收 Provider 的关闭通知 | 中 | ⏸️ 后续迭代 |

**实现说明**：
- Provider 在响应中携带 Closing 标记（通过 Response Attachment）
- Consumer Filter 检测响应中的 Closing 标记，标记该 Provider 为不可用

#### R6：移除不可用实例

**需求描述**：当感知到 Provider 下线后，将其从可用实例列表中移除。

**关键约束**：
- 移除后不应立即删除，需支持后续可能的恢复（如 Provider 重新上线）
- 需要与注册中心的实例管理机制配合

## 六、已知约束

1. **不支持的场景**：
   - 不支持在优雅下线过程中中止正在处理的请求
   - 不支持请求重定向到其他 Provider（由 Consumer 端决定）

2. **风险提示**：
   - 如果 Consumer 未及时响应通知，仍可能产生少量失败请求
   - 超时后的强制下线可能导致正在处理中的请求被中断
