# 优雅下线技术方案

## 1. 背景

远端 `upstream/develop` 当前只具备基础优雅下线能力：

1. 从注册中心摘流
2. 等待一小段时间，继续接收入站请求
3. 拒绝新请求并等待在途请求结束
4. 销毁协议资源

它不具备以下能力：

- provider 主动通知长连接 consumer
- consumer 收到通知后从本地实例目录删除当前 service 实例
- 长连接空闲时的快速切流

当前本地分支已经补了一部分能力：

- `Closing` 状态
- provider 响应 attachment 中带 `closing=true`
- consumer 侧 `closingInvokers`
- `grpc` / `triple` 的主动通知输入
- `RegistryDirectory` 的实例删除与 tombstone

但这些能力需要统一到一个语义清晰、阶段边界明确的 shutdown 流程中。

## 2. 目标

目标是实现以下能力：

1. provider 进入优雅下线后，能够尽快通知已建立长连接的 consumer
2. consumer 收到通知后，删除“当前地址下的当前 service 实例”
3. 即使没有新的业务响应返回，也能完成切流
4. 切流不依赖注册中心的实时性，但最终状态仍以注册中心事件为准
5. 第一阶段优先覆盖 `grpc` 和现代 `triple` 路径

## 3. 非目标

本方案第一阶段不解决以下问题：

- 所有协议一次性统一支持
- 一次删除整台地址上的所有 service
- 跨语言统一控制面标准
- 彻底替代注册中心
- 覆盖旧的 `Dubbo3Invoker` 兼容路径

## 4. 基本约束

### 4.1 删除粒度

consumer 侧删除的对象必须是：

- 当前地址下的当前 `service instance`

而不是：

- 这个地址上的所有 service

因此通知内容不能只包含 `Location`。

### 4.2 注册中心仍是最终事实来源

主动通知的作用是提前切流，不是替代注册中心。

因此本地删除必须满足：

- 注册中心删除事件到达后，状态自然收敛
- 注册中心尚未删除前，不会因为下一次 refresh 立刻把实例加回

### 4.3 目录层优先

真正决定后续负载均衡候选集的是 `RegistryDirectory`，不是 filter。

因此职责划分应为：

- filter 负责识别 closing 信号
- closing handler 负责统一处理事件
- directory 负责实例摘除与 tombstone
- registry refresh 路径负责 tombstone 判定

## 5. shutdown 主流程

收口后的 shutdown 主流程如下：

1. `mark closing`
2. `unregister registries`
3. `notify long connection consumers`
4. `wait provider inbound drain`
5. `wait consumer outbound drain`
6. `destroy protocols`
7. `execute custom callbacks`

### 5.1 第 2 步为什么必须是 unregister，而不是直接 destroy

第 2 步的目标是“先从注册中心摘掉”，不是“提前启动协议销毁”。

如果这里直接调用 `registry protocol.Destroy()`，会带来 4 个问题：

1. `Destroy()` 语义过重  
   当前 `registry/protocol` 的 `Destroy()` 不只是 `UnRegister`，还会启动延迟 `UnExport()`，并清理内部缓存。

2. 阶段职责混淆  
   第 2 步本应只做摘流；真正的协议销毁应在第 6 步。

3. 依赖内部实现细节  
   当前之所以“还能工作”，只是因为 `Destroy()` 内部恰好做了延迟 `UnExport()`。这不是稳定的 shutdown 契约。

4. 不利于后续演进  
   主动通知、tombstone、consumer 目录摘流都已经需要更清晰的阶段边界，第 2 步再继续绑定 `Destroy()` 会让流程越来越难推理。

因此当前实现改为：

- `graceful_shutdown` 优先调用可选接口 `protocolbase.RegistryUnregisterer`
- `registry/protocol` 实现 `UnregisterRegistries()`
- 只有旧实现不支持该接口时，才 fallback 到 `Destroy()`

这样可以兼顾：

- 新语义明确
- 旧行为兼容

## 6. 实例标识设计

### 6.1 结论

实例标识不应重新发明，应直接复用目录缓存 key。

代码调研结论：

- `registry.ServiceEvent.Key()` 默认使用 `Service.GetCacheInvokerMapKey()`
- `common.URL.GetCacheInvokerMapKey()` 当前由以下字段组成：
  - `protocol`
  - `username/password`
  - `ip:port`
  - `interface`
  - `group`
  - `version`
  - `timestamp`
  - `meshClusterID`

### 6.2 设计要求

通知消息中应包含：

- `instanceKey`
- `serviceKey`
- `address`

其中：

- `instanceKey` 是精确删除的主键
- `serviceKey` 和 `address` 主要用于路由、校验和日志

### 6.3 provider 侧来源

provider 在 export 阶段已经持有 `providerUrl`，可以直接计算：

```go
instanceKey := providerUrl.GetCacheInvokerMapKey()
```

因此 provider 不需要额外向注册中心查询实例唯一标识。

## 7. consumer 目录级摘除

### 7.1 为什么不能只依赖 `closingInvokers`

`closingInvokers` 只能做：

- filter 级请求拦截
- 短时间软屏蔽
- 本地兜底

它做不到：

- 从实例目录真正删除
- 刷新后续 `List()` 的结果
- 阻止注册中心 refresh 把实例加回

因此它不能作为最终方案。

### 7.2 正确落点

实例摘除应落在 `RegistryDirectory`。

原因：

- 实例真实缓存位于 `cacheInvokersMap`
- `setNewInvokers()` 会刷新 `cacheInvokers` 和 `RouterChain`
- 后续 `List()` 结果依赖目录和 router chain

### 7.3 目录层能力

目录层应暴露按 `instanceKey` 删除 closing 实例的能力：

```go
type ClosingInstanceRemover interface {
    RemoveClosingInstance(instanceKey string) bool
}
```

该能力的语义是：

1. 记录 tombstone
2. 按 `instanceKey` 从目录缓存删除实例
3. 刷新 `cacheInvokers`
4. 刷新 `RouterChain`
5. 返回是否命中并删除了实例

## 8. Closing Event Handler

### 8.1 设计目标

主动通知和被动通知都不应直接操作目录，而应统一转成 closing 事件。

建议入口：

```go
type ClosingEvent struct {
    InstanceKey string
    ServiceKey  string
    Address     string
    Source      string
}

type ClosingEventHandler interface {
    Handle(event ClosingEvent)
}
```

### 8.2 职责

`ClosingEventHandler` 负责：

- 根据 `serviceKey` 找到目标 directory
- 调用目录级删除能力
- 记录必要日志和指标

它不负责：

- 解析 attachment
- 判定错误是否属于 closing
- 直接和协议栈耦合

## 9. 被动通知

### 9.1 定位

provider 当前通过响应 attachment 带 `closing=true`。这条链路应保留，作为主动通知的补充。

### 9.2 正确升级方向

被动通知命中后，不应只做：

- `closingInvokers` 标记
- `SetAvailable(false)`

而应继续进入统一的 `ClosingEventHandler`，走目录级摘除。

## 10. 主动通知

### 10.1 目标

主动通知的目标不是“关闭连接”，而是：

- 在长连接空闲、没有业务响应返回时，也能让 consumer 尽快删除当前 service 实例

### 10.2 第一阶段协议范围

第一阶段覆盖：

- `grpc`
- 现代 `triple` 路径

不覆盖：

- `Dubbo3Invoker`

### 10.3 当前实现方向

当前主路径采用 `health watch` 作为 provider -> consumer 的主动通知输入：

- `grpc` server 注册 health service
- `triple` 复用现有 health watch
- provider 进入通知阶段后，把对应 service 状态置为 `NOT_SERVING`
- consumer watch 收到 `NOT_SERVING` 后，转成统一 `ClosingEvent`

这一步的关键点是：

- “通知”不再直接等同于 `GracefulStop()`
- 真正的 `GracefulStop()` / destroy 仍留在后续销毁阶段

## 11. tombstone 设计

### 11.1 要解决的问题

如果 consumer 本地已经删除了实例，但注册中心还没来得及完成删除：

- 下一次 registry refresh 仍可能把实例重新加回目录

### 11.2 核心方案

在 `RegistryDirectory` 内维护 tombstone 表：

- key: `instanceKey`
- value: 过期时间

处理流程：

1. 收到 closing 事件
2. 写入 tombstone
3. 删除目录实例
4. 后续 `doCacheInvoker()` 若命中 tombstone，则跳过实例重建

### 11.3 清理策略

采用混合策略：

- 注册中心删除事件到达时优先清理 tombstone
- 同时保留 TTL 兜底

不建议只靠 TTL，也不建议完全不设 TTL。

## 12. 方案选型

### 12.1 方案 A：继续扩展 `closingInvokers`

优点：

- 改动小

缺点：

- 不是目录级删除
- 无法阻止 registry refresh 复活
- 不能作为长期方案

结论：

- 不采用

### 12.2 方案 B：filter 直接操作 directory

优点：

- 表面链路短

缺点：

- 耦合严重
- filter 持有的是协议 invoker，不是稳定的 directory 引用
- 主动通知和被动通知难复用一套目录逻辑

结论：

- 不建议作为最终方案

### 12.3 方案 C：统一 closing 事件处理层

优点：

- 职责清晰
- 主动通知和被动通知可以复用
- 后续扩协议更自然

缺点：

- 需要引入一层事件处理

结论：

- 采用

## 13. 当前实现状态

已经完成：

- `RegistryDirectory` 的 `instanceKey` 级删除能力
- tombstone 机制
- `ClosingDirectoryRegistry` / `ClosingEventHandler`
- 被动通知和 closing 连接错误统一转为 closing 事件
- `grpc` / 现代 `triple` 的 health-watch 主动通知输入
- `notifyLongConnectionConsumers()` 并发 fan-out 和独立 timeout
- shutdown 第 2 步收口为 `unregister registries`

尚未完成：

- `Dubbo3Invoker` 覆盖
- 更完整的主动通知 ack 统计
- 端到端联调与更多协议覆盖

## 14. 实施顺序

1. 先实现目录级删除和 tombstone
2. 再统一 closing 事件入口
3. 接入被动通知
4. 接入主动通知
5. 收口 shutdown 主流程语义
6. 最后做联调、回归和扩覆盖

## 15. 风险点

1. provider 停服粒度和 consumer 删除粒度不一致  
   provider 偏向 address/server 级，consumer 目标是 service instance 级。

2. 注册中心 refresh 和本地 tombstone 并发冲突  
   需要保证 refresh 命中 tombstone 时稳定跳过。

3. 主动通知和被动通知同时命中  
   需要保证目录删除与 tombstone 写入幂等。

4. `grpc_protocol.go` 和 `triple.go` 改动风险高  
   必须优先保持现有业务路径稳定，新增逻辑尽量局部化并用定向测试覆盖。

## 16. 验收标准

1. provider 进入 graceful shutdown 后，consumer 能在注册中心事件到来前停止选中该实例
2. 删除粒度是“当前 service 实例”，不是整地址
3. registry refresh 不会立刻把刚删除的实例加回来
4. `grpc` 和现代 `triple` 的主动通知链路可用
5. shutdown 主流程阶段语义与代码实现一致
