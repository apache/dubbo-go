# 优雅下线技术方案

## 1. 背景

远端 `upstream/develop` 当前只具备基础优雅下线能力：

1. provider 反注册注册中心
2. 等待一段时间继续接收入站请求
3. 拒绝新请求并等待在途请求结束
4. 销毁协议资源

它不具备以下能力：

- provider 主动通知长连接 consumer
- consumer 收到通知后从本地实例目录删除当前 service 实例
- 长连接空闲时的快速切流

当前本地分支已经新增了一部分能力：

- `Closing` 状态
- provider 响应 attachment 中带 `closing=true`
- consumer 侧 `closingInvokers`
- `grpc` / `triple` 的 graceful shutdown callback

但这些实现距离需求仍有结构性差距。

## 2. 需求目标

目标是实现以下能力：

1. provider 进入优雅下线后，能够尽快通知已经建立长连接的 consumer
2. consumer 收到通知后，将“当前地址下的当前 service 实例”从本地实例目录删除
3. 即使没有新的业务请求往返，也能完成切流
4. 注册中心事件仍然是最终事实来源
5. 第一阶段仅覆盖 `grpc` 和 `triple`

## 3. 非目标

本方案第一阶段不解决以下问题：

- 所有协议统一一次性支持
- “整个地址上的所有 service 实例”一起摘除
- 跨语言统一控制协议标准化
- 完整的控制面治理体系

## 4. 当前实现存在的问题

### 4.1 远端 develop 的缺口

远端 `upstream/develop` 当前没有：

- `Closing` 状态
- 主动通知 consumer 的机制
- provider 被动通知 attachment
- consumer closing 感知和实例摘除

### 4.2 当前本地实现的主要问题

当前本地实现存在 4 个核心问题：

1. `notifyLongConnectionConsumers()` 实际执行的是 `GracefulStop()`，不是“主动通知”
2. `closingInvokers` 只是 filter 级软屏蔽，不是目录级删除
3. `closingInvokers` 无法阻止注册中心后续刷新把实例重新加回来
4. provider 当前更接近 address/server 级停服，而需求要求 consumer 按 service instance 级摘除

## 5. 基本约束

### 5.1 删除粒度

consumer 侧删除的对象必须是：

- 当前地址下的当前 `service instance`

而不是：

- 整个地址上的所有 service

这意味着通知消息不能只包含 `Location`。

### 5.2 注册中心仍是最终事实来源

主动通知的作用是提前切流，不是替代注册中心。

因此本地删除必须满足：

- 注册中心最终删除事件到达后，状态能够自然收敛
- 注册中心尚未删除前，不会因为下一次刷新把实例立刻加回来

### 5.3 目录层优先

真正决定负载均衡候选集的是目录层，而不是 filter 层。

因此整体方案应以目录层为主：

- filter 负责识别 closing 信号
- directory 负责摘除实例
- registry 刷新路径负责 tombstone 判定

## 6. 整体设计

### 6.1 总体分层

方案由四部分组成：

1. shutdown 主流程重构
2. provider -> consumer 主动通知
3. consumer 目录级实例摘除
4. tombstone 本地屏蔽机制

### 6.2 重构后的 shutdown 主流程

建议流程如下：

1. `mark closing`
2. `unregister registry`
3. `notify long connection consumers`
4. `wait provider inbound drain`
5. `wait consumer outbound drain`
6. `destroy protocols / close listeners`
7. `execute callbacks`

关键要求：

- “通知”阶段不能直接停 server
- `GracefulStop()` 必须放在真正 destroy 阶段

## 7. 实例标识设计

### 7.1 结论

当前实例标识不应重新发明，应直接复用现有目录缓存 key。

代码调研结论：

- `registry.ServiceEvent.Key()` 默认使用 `Service.GetCacheInvokerMapKey()`
- `common.URL.GetCacheInvokerMapKey()` 当前由以下字段组成：
  - `protocol`
  - `username/password`
  - `ip:port`
  - `interface`
  - `group`
  - `version`
  - `timestamp`
  - `meshClusterID`

### 7.2 设计要求

主动通知消息中必须包含：

- `instanceKey`
- `serviceKey`
- `address`

其中：

- `instanceKey` 是精确删除的主键
- `serviceKey` 和 `address` 用作辅助校验与日志观测

### 7.3 provider 侧来源

provider 在 export 阶段已经持有 `providerUrl`，可以直接计算：

```go
instanceKey := providerUrl.GetCacheInvokerMapKey()
```

因此 provider 不需要额外查询注册中心来获得实例唯一标识。

## 8. consumer 目录级摘除设计

### 8.1 为什么不能只依赖 closingInvokers

`closingInvokers` 的作用仅限于：

- filter 级请求拦截
- 短时间软屏蔽
- 本地临时兜底

它做不到：

- 从实例目录真正删除
- 更新后续 `List()` 的结果
- 阻止注册中心刷新把实例重新带回

因此它不能作为最终方案。

### 8.2 正确落点

实例摘除的核心应落在 `RegistryDirectory`。

理由：

- 实例真实缓存位于 `cacheInvokersMap`
- `setNewInvokers()` 会刷新 `cacheInvokers` 与 `RouterChain`
- `List()` 最终依赖目录与路由链输出

### 8.3 目录层需要具备的能力

目录层需要暴露一个“按 instanceKey 删除 closing 实例”的能力。

建议形式：

```go
type ClosingInstanceRemover interface {
    RemoveClosingInstance(instanceKey string) bool
}
```

该能力的语义是：

1. 按 `instanceKey` 从目录缓存删除目标实例
2. 刷新 `cacheInvokers`
3. 刷新 `RouterChain`
4. 返回是否命中并删除了实例

### 8.4 为什么不建议直接在 filter 里硬连 directory

这是一个关键设计点。

consumer filter 在调用链里拿到的是“协议 invoker”，而不是 `RegistryDirectory`。因此不能直接通过简单断言拿到目录对象。

如果强行在 filter 层直接操作目录，会出现几个问题：

1. filter 和具体 directory 实现强耦合
2. `grpc` / `triple` / cluster 包装层会把调用链搞得很脆
3. 后续主动通知与被动通知难以复用同一条目录逻辑

因此更合理的设计是：

- filter 只识别 closing 信号
- 把 closing 事件交给一个 consumer-side instance state manager
- 由该 manager 再驱动对应的 directory 实例删除

## 9. 被动通知设计

### 9.1 现有 attachment 方案的定位

provider 当前通过响应 attachment 附带 `closing=true`。

这条链路不应废弃，而应保留为补充手段。

### 9.2 正确的升级方向

被动通知命中后，不应只做：

- `closingInvokers` 标记
- `SetAvailable(false)`

而应进入与主动通知一致的“目录级摘除”逻辑。

### 9.3 推荐架构

建议引入一个统一的 closing 事件入口：

```go
type ClosingEventHandler interface {
    HandleClosingInstance(instanceKey string, serviceKey string, address string)
}
```

然后：

- 被动通知 attachment 命中后，调用该入口
- 主动通知 RPC 收到后，也调用该入口

这样可以避免两套删除逻辑。

## 10. 主动通知设计

### 10.1 目标

主动通知的目标不是“关闭连接”，而是：

- 在长连接空闲、没有业务请求返回时，也能尽快让 consumer 删除当前 service 实例

### 10.2 通知内容

建议统一定义：

```go
type ClosingNotice struct {
    Protocol    string
    ServiceKey  string
    Address     string
    InstanceKey string
    Timestamp   int64
}
```

### 10.3 确认机制

consumer 收到通知后返回 ack，语义为：

- 已完成本地实例目录摘除处理

provider 在通知阶段统计：

- 通知目标数
- 成功 ack 数
- 失败数
- 超时数

### 10.4 协议范围

第一阶段只做：

- `triple`
- `grpc`

### 10.5 当前代码调研结论

`triple` 和 `grpc` 都具备流式 RPC 能力，但仓库内当前没有现成的通用 provider -> consumer closing 控制通道。

因此第一阶段需要新增：

- 内部控制 RPC
或
- 内部控制 stream

不能把现有 `GracefulStop()` 当作主动通知。

## 11. tombstone 设计

### 11.1 要解决的问题

如果 consumer 本地删除了实例，但注册中心还没有完成反注册：

- 下一次 registry refresh 仍可能把该实例重新加回目录

这会导致刚摘除的实例复活。

### 11.2 核心设计

在 `RegistryDirectory` 内维护 tombstone 表：

- key: `instanceKey`
- value: 过期时间 / 状态信息

处理流程：

1. 收到 closing 事件
2. 从目录删除实例
3. 记录 tombstone
4. 后续 registry refresh 到达时，若命中 tombstone，则忽略该实例

### 11.3 清理策略

建议第一阶段采用混合策略：

- 注册中心删除事件到达时提前清理 tombstone
- 同时保留 TTL 兜底清理

不建议：

- 只靠 TTL
- 或完全不设 TTL

## 12. 方案选型对比

### 12.1 方案 A：只扩展 closingInvokers

思路：

- 继续增强 filter 层
- 收到 closing 后只做本地软屏蔽

优点：

- 改动最小

缺点：

- 不是目录级删除
- 容易被上层重选
- 无法解决注册中心刷新复活问题

结论：

- 不采用

### 12.2 方案 B：filter 直接操作 directory

思路：

- consumer filter 直接找到并修改 `RegistryDirectory`

优点：

- 表面上链路最短

缺点：

- 耦合严重
- 难以跨 cluster / protocol 复用
- 目录定位不稳定

结论：

- 不建议作为最终架构

### 12.3 方案 C：引入统一 closing 事件处理层

思路：

- 主动通知和被动通知都转成统一 closing 事件
- 由 consumer-side manager 负责目录摘除和 tombstone

优点：

- 职责清晰
- 主动 / 被动通知可复用
- 后续扩协议更自然

缺点：

- 第一阶段改动面稍大

结论：

- 推荐采用

## 13. 分阶段实施建议

### 阶段 1：基础能力

1. 固化 `instanceKey` 设计
2. 在 directory 层定义 closing instance 删除能力
3. 在 `RegistryDirectory` 实现目录级删除

### 阶段 2：状态管理

1. 引入 consumer-side closing 事件处理层
2. 接入被动通知 attachment
3. 增加 tombstone

### 阶段 3：主动通知

1. 设计 `grpc` 主动通知通道
2. 设计 `triple` 主动通知通道
3. 让主动通知复用同一套 consumer closing 事件处理逻辑

### 阶段 4：shutdown 流程收敛

1. 重构 shutdown 主流程
2. 把“通知”和“停服”彻底拆开
3. 补齐统计、超时、重试

## 14. 风险点

### 14.1 粒度风险

provider 当前很多管理逻辑偏 address/server 级，而需求要求 consumer 按 service instance 级删除。

如果不统一，容易出现误删或误停。

### 14.2 注册中心竞态

没有 tombstone 时，本地删除会被注册中心刷新覆盖。

### 14.3 协议能力风险

`grpc` / `triple` 虽然支持流式 RPC，但不代表已经有可复用的控制面通道。

### 14.4 被动通知覆盖率风险

attachment 方案依赖正常响应返回。错误路径、单向调用、流式场景不一定都覆盖。

## 15. 验收标准

满足以下条件视为第一阶段方案完成：

1. `grpc` / `triple` provider 进入优雅下线后，空闲长连接 consumer 能收到 closing 通知
2. consumer 能删除“当前地址下的当前 service 实例”
3. 同地址其他 service 不受影响
4. 注册中心未及时推送时，本地 tombstone 能阻止实例复活
5. 注册中心最终删除事件到来后，本地状态正确收敛
6. 被动通知仍可作为补充生效

## 16. consumer-side closing handler 详细设计

### 16.1 设计目标

这一层的职责是把不同来源的 closing 信号统一收敛成一套本地处理逻辑。

它要解决的问题是：

1. 主动通知和被动通知不能各自直接改目录
2. filter 识别到 closing 后，不应直接依赖 `RegistryDirectory` 具体实现
3. tombstone 写入、目录删除、重复通知幂等，需要集中处理

因此需要一层独立的 consumer-side closing handler。

### 16.2 输入来源

closing handler 的输入至少有两类：

1. 被动通知  
provider 正常响应返回 attachment：`closing=true`

2. 主动通知  
provider 通过 `grpc` / `triple` 内部控制 RPC 或控制 stream 发送 `ClosingNotice`

未来如果需要，也可以扩展第三类：

3. 本地连接错误推断  
例如 `transport is closing`

但第一阶段不建议让“连接错误推断”直接驱动目录删除，它更适合作为 filter 级兜底。

### 16.3 handler 接口建议

建议在 graceful shutdown 领域层定义统一入口，例如：

```go
type ClosingEvent struct {
    Source      string
    InstanceKey string
    ServiceKey  string
    Address     string
}

type ClosingEventHandler interface {
    HandleClosingEvent(event ClosingEvent) bool
}
```

语义要求：

- 返回值表示是否命中了本地目录并成功触发处理
- `InstanceKey` 是主键
- `ServiceKey` 和 `Address` 用于辅助校验、日志和后续索引优化

### 16.4 handler 的处理流程

`HandleClosingEvent` 内部建议按以下顺序执行：

1. 参数校验  
`InstanceKey` 不能为空

2. 幂等判断  
如果该 `instanceKey` 已经处于 closing/tombstone 状态，则直接返回

3. 写入 tombstone  
先写 tombstone，再删目录，避免删除和 registry refresh 并发时出现短窗口复活

4. 从目录删除实例  
通过目录能力接口按 `instanceKey` 删除实例

5. 记录结果和日志  
包括来源、是否命中、命中了哪个 service、是否已经是 closing

### 16.5 为什么先写 tombstone 再删目录

如果先删目录，再写 tombstone，会有一个竞态窗口：

1. 目录刚删掉实例
2. tombstone 还没写入
3. 注册中心刷新正好进来
4. 实例被重新加回

因此顺序应该是：

1. 先让“后续 refresh 不得重建”这个条件成立
2. 再删除当前目录中的实例

## 17. tombstone 详细设计

### 17.1 存放位置

tombstone 必须放在 `RegistryDirectory`，而不是 filter 或全局 graceful shutdown filter 中。

原因：

1. 它要拦截的是目录刷新路径
2. 它和 `cacheInvokersMap` / `doCacheInvoker()` 强相关
3. 同一个 consumer 进程可能存在多个 service directory，状态应跟随目录实例

### 17.2 数据结构建议

建议结构如下：

```go
type closingTombstone struct {
    InstanceKey string
    ServiceKey  string
    Address     string
    Source      string
    ExpireAt    time.Time
}
```

`RegistryDirectory` 内建议增加：

```go
tombstones sync.Map // map[string]closingTombstone
```

key 直接使用 `instanceKey`。

### 17.3 基本操作

需要以下 4 个基本动作：

1. `markClosing(instanceKey, meta)`  
写入或刷新 tombstone

2. `isClosing(instanceKey)`  
判断 tombstone 是否仍然有效

3. `clearClosing(instanceKey)`  
收到注册中心删除事件后删除 tombstone

4. `cleanupExpiredClosing()`  
按 TTL 清理过期 tombstone

### 17.4 接入点

调研后最关键的 3 个接入点是：

1. `RemoveClosingInstance(instanceKey)`  
用于 closing handler 驱动的本地删除

2. `doCacheInvoker(newUrl, event)`  
用于拦截注册中心刷新带来的实例重建

3. `uncacheInvoker(event)` 或 delete 事件处理路径  
用于注册中心确认删除后清理 tombstone

### 17.5 与 refresh 流程的配合

建议规则如下：

1. registry add/update 事件到达
2. 在 `doCacheInvoker()` 前先计算 `event.Key()`
3. 若该 key 命中有效 tombstone：
   - 跳过 `Refer`
   - 跳过 `cacheInvokersMap.Store`
   - 记录 debug/info 日志
4. 若未命中：
   - 按当前逻辑正常重建 invoker

### 17.6 与 delete 事件的配合

当 registry delete 事件到达时：

1. 按现有逻辑删除目录缓存
2. 同时删除该 `instanceKey` 对应的 tombstone

这表示：

- 本地 closing 状态已经被注册中心最终事实确认
- 后续不再需要 tombstone 阻止复活

### 17.7 TTL 策略

推荐第一阶段采用：

- delete 事件优先清理
- TTL 兜底清理

TTL 的作用不是替代 delete 事件，而是解决异常情况：

1. provider 崩溃后通知发出但 delete 事件长期没来
2. 注册中心抖动导致 delete 丢失或延迟
3. 避免 tombstone 永久积累

### 17.8 幂等要求

tombstone 必须支持幂等：

1. 同一实例被重复通知多次，不应重复删除出错
2. 主动通知和被动通知同时到达，结果应一致
3. delete 事件到达时，即使目录中早已没有该实例，也应能安全清理 tombstone

## 18. directory 与 closing handler 的连接方式

### 18.1 不推荐的方案

不推荐直接在 consumer filter 里：

- 遍历全局 directory
- 或直接硬编码查找 `RegistryDirectory`

原因是：

1. 耦合重
2. 调用链不稳定
3. 主动通知和被动通知无法优雅复用

### 18.2 推荐方案

推荐由 graceful shutdown 领域层维护一个“directory 能力注册中心”，但它不直接暴露给 filter 做任意操作，而只服务于 closing handler。

建议形式：

```go
type ClosingDirectoryRegistry interface {
    Register(serviceKey string, remover ClosingInstanceRemover)
    Unregister(serviceKey string, remover ClosingInstanceRemover)
    Find(serviceKey string) []ClosingInstanceRemover
}
```

这里注册维度至少应包含 `serviceKey`，而不是全局广播。

### 18.3 为什么要按 serviceKey 建索引

如果只做全局 remover 列表广播，会有这些问题：

1. 每次 closing 都要 fan-out 给所有目录
2. 多 service 场景下效率差
3. 日志和命中关系不清晰
4. 后续主动通知里已经有 `serviceKey`，不用掉这个信息会浪费

因此推荐：

- directory 创建时按 `serviceKey` 注册 remover
- closing handler 处理时，先按 `serviceKey` 找目标目录
- 再按 `instanceKey` 做精确删除

### 18.4 serviceKey + instanceKey 的处理顺序

handler 收到事件后的建议顺序是：

1. 先按 `serviceKey` 过滤目录集合
2. 再在目标目录里按 `instanceKey` 精确删除

这样既避免全局广播，也避免“只按 serviceKey 粗删”。

## 19. 被动通知如何接入 closing handler

### 19.1 现状

当前 `consumer_filter.OnResponse()` 命中 `closing=true` 后，只会：

1. 标记 `closingInvokers`
2. 设 invoker 为 unavailable

### 19.2 目标改造

改造后应变为：

1. 识别 `closing=true`
2. 从 `invoker.GetURL()` 提取：
   - `instanceKey`
   - `serviceKey`
   - `address`
3. 调用统一 `ClosingEventHandler`
4. 保留 `closingInvokers` 作为 filter 级短期兜底

### 19.3 为什么 closingInvokers 仍然要保留

即使有目录级删除，`closingInvokers` 仍然有价值：

1. 目录删除前的短窗口兜底
2. closing handler 执行失败时的本地保护
3. 对连接错误推断场景的短时规避

但它的定位必须降级为“兜底”，不是主方案。

## 20. 下一步实现建议

在代码实现上，建议下一步只做以下内容：

1. 在 `RegistryDirectory` 中增加 tombstone 数据结构和基本操作
2. 在 `doCacheInvoker()` 中接入 tombstone 判定
3. 在 delete 事件路径接入 tombstone 清理
4. 定义 `ClosingEventHandler` 和 `ClosingDirectoryRegistry` 接口

暂时不要做：

1. `grpc` / `triple` 主动通知协议实现
2. shutdown 主流程重构
3. 连接错误直接驱动目录删除

原因是前两者都依赖 closing handler + tombstone 先稳定下来。
## 21. Active Notice Phase-1 Implementation Choice

The phase-1 active notice path is implemented with `health watch`, not a new custom control RPC.

Why:

1. provider cannot push a new message to an idle consumer over a normal unary connection
2. both `grpc` and `triple` already support long-lived server-stream semantics
3. `triple` already has an internal `grpc.health.v1.Health/Watch` implementation
4. `grpc` can reuse the standard gRPC health service and `Watch`

Provider-side behavior:

1. during `notifyLongConnectionConsumers()`, `grpc` / `triple` publish `NOT_SERVING`
2. this stage must not call `GracefulStop()`
3. real server stop still happens in the later destroy stage

Consumer-side behavior:

1. `Refer()` starts a health watch goroutine for the current invoker
2. the watch target is the current `serviceKey`
3. when `NOT_SERVING` is received, the consumer emits a unified `ClosingEvent`
4. the existing `ClosingEventHandler -> RegistryDirectory tombstone` path handles the rest

Current limitation:

- the phase-1 triple active watch is wired on the modern `TripleInvoker` path
- the legacy `Dubbo3Invoker` path is not covered yet and still falls back to passive notice / connection-error handling
